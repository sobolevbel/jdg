# Руководство контрибьютора

Спасибо, что хотите улучшить гайд! Вопросы можно задать в
[чате контрибьюторов](https://t.me/+WK5ppqgHxXA3MjAy).

## Быстрая правка (без git)

Для опечаток и мелких правок достаточно браузера:

1. Нажмите карандаш ✏️ в шапке страницы гайда — откроется веб-редактор GitHub.
2. Поправьте текст, нажмите **Commit changes** — GitHub сам создаст форк и Pull Request.

## Полный цикл (для крупных правок)

1. Форкните репозиторий и создайте ветку.
2. Внесите изменения, проверьте локально, что сайт собирается (см. ниже).
3. Запушьте ветку и создайте Pull Request в `master`.

Пошагово: [Creating a pull request](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/proposing-changes-to-your-work-with-pull-requests/creating-a-pull-request)
(EN) или [Внесение собственного вклада в проекты](https://git-scm.com/book/ru/v2/GitHub-%D0%92%D0%BD%D0%B5%D1%81%D0%B5%D0%BD%D0%B8%D0%B5-%D1%81%D0%BE%D0%B1%D1%81%D1%82%D0%B2%D0%B5%D0%BD%D0%BD%D0%BE%D0%B3%D0%BE-%D0%B2%D0%BA%D0%BB%D0%B0%D0%B4%D0%B0-%D0%B2-%D0%BF%D1%80%D0%BE%D0%B5%D0%BA%D1%82%D1%8B) (RU).

## Локальная сборка

Обычный способ (Python 3):

```shell
pip install -r requirements.txt
python -m mkdocs serve
```

Docker:

```shell
docker compose up --build
```

Сайт поднимется на <http://localhost:8000> и будет перезагружаться при правках.

## Соглашения по оформлению страниц

- **Frontmatter** в начале каждой страницы: `title` (используется в SEO),
  `description` (описание для поисковиков), `tags`.
- Один заголовок `#` (H1) на страницу; в заголовках нет завершающих знаков
  препинания (правило линтера MD026).
- Выноски — через [admonitions](https://squidfunk.github.io/mkdocs-material/reference/admonitions/):
  `!!! info`, `!!! warning`, `!!! danger`.
- **Ссылки** — reference-style, нумерованные по порядку появления в тексте:
  `[текст][1]`, определения `[1]: https://…` внизу файла.
- Текст ссылки должен описывать, куда она ведёт: не «подробнее [тут]», а
  «подробнее в [инструкции по заполнению ZUS DRA]». Скринридеры зачитывают
  ссылки списком вне контекста, а поисковики учитывают анкорный текст.
- В конце страницы — кнопка «Поддержите наш гайд чашкой кофе ♥».
- Заметная правка? Добавьте строку в [docs/whatsnew.md](docs/whatsnew.md)
  (и в `whatsnew.en.md`).

## Скриншоты

- Складывайте в `docs/images/<тема>/` рядом с остальными картинками страницы.
- Единый стиль рамки — через [screely.com](https://screely.com/):
  Window Type **Plain Window**, Window Style **Regular**, Background Color
  **белый**, Padding Vertical/Horizontal **4px**.
- Персональные данные (NIP, PESEL, номера счетов, суммы) перед публикацией
  замазывайте или блюрьте.
- **Alt-текст обязателен и должен быть осмысленным**: что видно на экране и
  какой элемент важен. Не имя файла, а описание:

  ```markdown
  ![Форма RWN: раздел Dane wniosku, кнопка Dalej][5]
  ```

  CI не пропустит картинки с пустым alt, именем файла или голым числом.

## Английская версия

У сайта два языка: русский оригинал (`file.md`) и английский перевод
(`file.en.md`) в той же папке.

- Правите русскую страницу — **обновите английскую** в том же PR. Если не
  владеете английским, напишите в описании PR «нужен перевод» — переведём.
- Новая страница: создайте `file.md` + `file.en.md`, добавьте страницу в `nav:`
  в `mkdocs.yml` и её перевод в `nav_translations` (секция плагина `i18n`).
- Якоря заголовков в RU и EN различаются (транслитерация против английских
  слагов) — проверяйте внутренние ссылки на якоря в обеих версиях.

## Проверки перед PR

CI прогоняет линтер, проверку ссылок и сборку. Локально:

```shell
# линтер markdown (gem install mdl)
mdl -s .github/workflows/markdown_linter_rules.rb docs/.

# сборка без ворнингов
python -m mkdocs build
```
